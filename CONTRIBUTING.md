# 👥 Development guide

This project is for personal use and is not open to public contributions. This document describes how to develop and test Dev Container Features locally for future reference.

## 🛠️ Tech stack

This project uses the following tech stack:

- [Bash](https://www.gnu.org/software/bash/)
- [Dev Container CLI](https://github.com/devcontainers/cli)

## 📂 Key directory structure

- `src/`: Source of each feature
- `test/`: Tests for each feature
  - `test/_global`: Tests for global scenarios
- `flake.nix`: Flake configuration for development environment
- `Justfile`: Commands for development

## 🔧 Set up development environment

For development, you need to have the following tools installed:

### 🐋 System requirements

You need to have [Docker](https://www.docker.com/) installed. Docker cannot be installed via Nix Flakes, so install it manually on your host machine.

### ❄️ Tools managed via Nix Flakes

This repository uses [Nix Flakes](https://nix.dev/concepts/flakes.html) to manage tools. The following tools will be automatically installed (you need `nix` installed):

- `pre-commit`
- `just`
- `devcontainer`
- `shellcheck`
- `shfmt`

Run `nix develop` to start the development environment, then run `just install` to install dependencies.

We also provide a Dev Container configuration ([devcontainer.json](./.devcontainer.example/devcontainer.json)) that sets up Nix and Docker (DinD).

## ⌨️ Adding a new feature

You can add a new feature as follows:

- Copy an existing feature in the `src/` directory as a template, then modify it
- Add tests in the `test/` directory
- Run `just test <feature-id>` to test the feature

New features are automatically detected by the automation workflow; you don't need to update any configuration file manually.

> [!NOTE]
> You don't need to make the documentation complete right away. Documentation is automatically generated by the release workflow; if you want, you can autogenerate a draft by running `devcontainer features generate-docs --namespace lasuillard/devcontainer-features` at the project root.

## ✅ Verifying changes

Before you push your code, verify that it follows the project's coding standards. Run `just ci` to run the linters, formatters, and some tests.

Because testing features can be time-consuming, only specific features are tested locally. The CI workflow runs tests for all features.

## 🚀 Releasing features

When changes are pushed to the repository's `main` branch and a feature's `devcontainer-feature.json` file is updated, the feature is released automatically via the [release.yaml](./.github/workflows/release.yaml) workflow.
